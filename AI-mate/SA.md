# 系統架構 (SA)
## 系統設計層面 (SD 問題)

### 系統前後台技術
- **前台**: 使用 HTML (網頁技術)。
- **後台**: 雖然來源資料未明確指定，但基於業務需求（庫存、會員、支付、活動追蹤），**系統必須有一個強大的後台（Backend）** 來提供 API 服務並連接資料庫。推薦使用 Spring Boot。

### 資料來源：使用者輸入 vs. 自動產生

| 資料類型 | 應由使用者輸入 | 應自動產生 |
|---|---|---|
| **訂單相關** | 飲品選擇、數量、甜度/冰度、加料選擇。 | 訂單編號、訂單時間、總金額、是否滿足扭蛋門檻。 |
| **行銷/會員** | 會員資料（姓名、聯繫方式）、參與「拍照打卡挑戰」的上傳內容。 | 扭蛋活動的隨機獎項（折價券）、會員積分、活動觸發狀態。 |
| **產品資訊** | （無，產品資訊由後台維護） | （無） |

### 前後端職責劃分

| 介面 | 顯示/呈現的資料 (Front-end) | 後台提供的支援 (Back-end) |
|---|---|---|
| **產品頁** | 「金蛋奶茶」韓文/中文文案、核心賣點、價格、選項（如甜度可調）。 | 提供 `/menu` API 查詢產品列表；提供 `/product/{id}` API 查詢產品詳情及庫存。 |
| **活動頁** | **「金蛋幸運日」**的活動日期（週一）、扭蛋機活動的門檻（滿 120 元）。 | 提供 `/promotion` API 查詢當前活動狀態；提供 `/loyalty` API 查詢會員的優惠券。 |
| **結帳頁** | 訂單總金額、可使用的優惠券列表。 | 提供 `/order/create` API 處理訂單建立，包含庫存檢查和優惠券應用。 |

### API 命名優化建議
為了讓 API 名稱與品牌「雞娃娃」和其「金蛋」的意象連結，可以採用具備療癒或動物主題的命名：

| 原始功能 (Hypothetical) | 建議簡潔 API 名稱 | 命名理由 |
|---|---|---|
| `GET /products` | `/chick/menu` (小雞菜單) | 呼應可愛小雞吉祥物。 |
| `POST /orders` | `/egg/hatch` (孵化訂單) | 呼應「金蛋奶茶」的「幸福就在嘴裡孵化」文案。 |
| `GET /promotions/gacha` | `/treasure/spin` (寶藏旋轉) | 呼應「金蛋象徵驚喜與幸福」的寶藏概念。 |
| `POST /users/checkin` | `/nest/luckyday` (巢穴幸運日) | 專門用於處理「金蛋幸運日」的打卡回訪。 |

### 未來擴充性
從行銷文案和活動設計中，可以高度推測品牌已經考慮到幾個方面的擴充性：
1. **國際化 (i18n)**： 系統和內容設計具備擴充性，能支援多語言。來源資料中提供了完整的韓文版產品文案和社群標籤，顯示品牌在產品命名和行銷溝通層面已為進入韓國或其他市場做準備。
2. **產品類別擴充**： 扭蛋機活動提供「周邊」作為獎勵。這暗示產品管理模組可能已經設計成不僅限於飲品（如「咕咕奶茶」），還可以擴充銷售非飲品的實體商品（如雞娃娃玩偶）。
3. **活動多樣化**： 設計了三種獨立且目標各異的活動模듈（回訪率、客單價、UGC），未來可以輕易地增加季節限定、聯名合作或其他新的促銷規則。

### 系統比喻
這個數位系統就像是為一隻活潑的「雞娃娃」建造的**「數位雞舍」**。前台是漂亮可愛、充滿打卡點的展示區（「萌」視覺導向），吸引客人停留；而後台則是堅固且自動化的飼料機和孵蛋器，它確保了「金蛋奶茶」的布丁、珍珠等核心物料供應無虞，並能精準地在週一自動派發「幸福的飼料」（加布丁優惠），讓可愛的雞群（顧客）不斷回訪。
